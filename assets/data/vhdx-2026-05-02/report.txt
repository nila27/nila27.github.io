==============================================================================
  1. EXECUTIVE SUMMARY
==============================================================================

Three findings dominate the data:

  (1) FIRST-WRITE PENALTY is the only large differentiator between formats.
      VHDX-Dynamic suffers a ~950x slowdown on first writes due to its 32 MiB
      default block size. VHD-Dynamic and VHDX-Differencing (both with 2 MiB
      blocks) are 15x faster on the same workload.

  (2) STEADY-STATE performance is nearly identical across all 5 formats.
      Once data is allocated, BAT and parent-chain costs amortize to noise.
      Format choice matters for storage shape, NOT for runtime performance
      after warm-up.

  (3) CPU cost per IO is also nearly identical across formats. The user-mode
      I/O path (CreateFile + IOCP + WriteFile) dominates kernel-side vhdmp
      overhead at our IOPS levels. CPU efficiency analysis (Section 6) shows
      <10% variation across formats for any given workload.

==============================================================================
  2. METHODOLOGY
==============================================================================

Test setup per iteration:
  - 4 GiB virtual disk, 2 GiB IO range
  - Fresh disk created and attached at iteration start
  - All disks deleted at iteration end (next iteration creates fresh)

Phases run per format:
  - first_write : 4K seq write QD1, stride-aligned, touches each block twice
                  (only on thin formats: VHD-Dynamic, VHDX-Dynamic, Differencing)
  - prealloc    : 1 MiB seq write fills IO range (thin formats only)
  - matrix      : 24 cells = (4K, 64K, 1M) x (QD1, QD32) x (seq, rand) x (R, W)
  - mixed_rw70  : random 30% writes / 70% reads, QD32, 4K and 64K, 5s time-bounded
  - hotspot     : random 80% IO in 20% of range, QD32, 4K and 64K, 5s
  - stride      : 4K block on 64K stride sequential, QD1+QD32, 5s
  - rewrite     : 4K QD32 sequential write to 16 MiB region, 5s (steady-state)

Cooldown between iterations:
  - SetSystemFileCacheSize hard-trim (NTFS Cache Manager flush)
  - 1 GiB memory pressure (VirtualAlloc + page touch + VirtualFree)
  - 120 second sleep

Block size hypothesis (default values for each format):
  - VHD-Dynamic:        2 MiB block -> small per-allocation cost
  - VHDX-Differencing:  2 MiB block -> same
  - VHDX-Dynamic:      32 MiB block -> 16x larger allocation per first-touch

The 32 MiB default for VHDX-Dynamic is widely considered correct for guest VM
use, where workloads do not start from a blank disk. For host-side analytics
or large file initialization scenarios, override with BlockSizeInBytes when
calling CreateVirtualDisk.

==============================================================================
  9. CAVEATS AND LIMITATIONS
==============================================================================

  - VHDMP IRP event capture failed: the Microsoft-Windows-VHDMP provider
    routes per-IO events (VHD_START_IO / VHD_COMPLETE_IO) only to the
    Analytic channel's .evtx log, not to ETW real-time sessions enabled
    via the standard WPR profile path. So the ETL traces do not contain
    per-IRP amplification data. This was not blocking; effective
    amplification was derived from IOPS / latency ratios instead.

  - Read tests in 2 GiB IO range fit entirely in OS page cache on this
    machine. The 'hotspot' phase therefore showed no benefit over uniform
    random. To exercise true cache pressure, IO range must exceed RAM.

  - SSD-specific results: numbers reflect this particular drive's behavior.
    Latency p99 spikes (1M QD32 writes hit 100ms+) are SSD-side write
    throttling, not vhdmp behavior.

  - First-write penalty depends on default block size. VHDX-Dynamic was
    tested with the API default (32 MiB). Specifying a smaller value via
    CreateVirtualDisk's BlockSizeInBytes parameter would close the gap.

  - Single-machine, single-drive measurement; no replication.

==============================================================================
  10. RECOMMENDATIONS
==============================================================================

By use case:

  Maximum performance, predictable latency:
      VHD-Fixed or VHDX-Fixed (no first-write penalty, identical steady state)
      Storage cost: full disk size up front.

  Sparse storage with new-disk performance:
      VHD-Dynamic (2 MiB block) -- accepts ~63x first-write penalty in
      exchange for thin-provisioned size.

  VHDX features (>2 TB, 4K alignment, log-based corruption resistance):
      VHDX-Dynamic with explicit BlockSizeInBytes = 2 MiB or 4 MiB.
      Default 32 MiB block makes first writes catastrophically slow.

  Layered/snapshot scenarios:
      VHDX-Differencing -- cheap snapshots from a shared parent.
      Steady-state performance equal to other formats.

  Block size override at creation:
      CreateVirtualDisk(..., params.Version2.BlockSizeInBytes = 2*1024*1024)
      Reduces VHDX-Dynamic's first-write penalty by ~16x (matches VHD-Dynamic).
